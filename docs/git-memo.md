## ■ Git管理の考え方

### ① コミット履歴 = 家系図（親 → 子 → 孫）
Git の履歴は次のように 一本の血統（直線） でつながっている必要がある。
```Repository
A（親）
 ↓
B（子）
 ↓
C（孫）
```
ローカルとリモートがこの家系図を共有していれば、push は問題なく通る。


### ② 履歴がズレた家系図（push が拒否される状態）
共同作業者が先に push していた場合、ローカルはリモートの最新コミットを知らないことがある。
```リモート
A → B → C
```
```ローカル
A → B → D
```
Git の判断：
「B の子は C である家系図が正しいのに、
B の子が D という別の孫が存在する家系図になっている。
家系図が壊れるから push は拒否する。」
=> non-fast-forward（履歴不一致）。


### ③ push が拒否されたら pull して「孫を知る」必要がある
push が拒否された場合、ローカルはリモートの最新（C）を知らない状態。
そのため、まず pull して C を取り込み、家系図をつなげる。

```孫を取り込んで、ひ孫として家系図を繋げる
リモート: A → B → C
ローカル: A → B → D   （C を知らない）
↓
pull 後:
A → B → C → D'   （D' は C を親にした新しいコミット）
※ D は「C を親にした新しい自分のコミット（ひ孫）」
※ 自動マージできない場合はコンフリクトが発生し、手修正が必要
```

### ④ コンフリクトが起きる条件
Git が pull したとき、C と D を自動マージして、
現在の変更に最新を取り込んだ状態にしようとする。
この時、変更行が被っていなければ自動マージできる（コンフリクトは発生しない）が、
同じ行を変更していることが発覚すると、C と D のどちらを優先するのか
Git は判断できない。
ローカル側からすると、Cで同じ個所を変更していたことを知らないから、
同時に差し出されたような体裁になる。

コンフリクトが発生しないパターン
```同じファイルの変更履歴
リモート： A → B → C(2,3行目を変更)
ローカル： A → B → D(5,6,7行目を変更)
↓
pull で D' (2,3,5,6,7) に自動マージできる
```

コンフリクトが発生するパターン
Git は “同じ行” を両方が変更したときだけコンフリクトする
```同じファイルの変更履歴
リモート： A → B → C(2,3行目を変更)
ローカル： A → B → D(3,4,5行目を変更)
↓
pull で D' (2,3,4,5) にマージしたいところだが、
3行目の変更が衝突するため自動マージできない。（手修正が必要）
```


### ⑤ push は「上書き」ではなく「履歴を追加する」操作
push はリモートの履歴を上書きするのではなく、
リモートの家系図の末尾に、自分の子孫を追加する操作である。
そのため、履歴がつながっていない場合は push を拒否する。

### ⑥ 強制 push（git push -f）が必要になるケース
git push -f origin main
基本的に使わないが、以下のような場合は必要になる：
リポジトリ作成直後の README だけを消してローカルを正にしたい
rebase 後の履歴をリモートに反映したい
誤って push したコミットを消したい
※ 強制 push は他人の履歴を上書きしてしまうため、共同作業では絶対に使わない。

## ■ Git コマンド

### Git側 リポジトリ作成(web)
Gitのpushトークンは再確認できないので保存必須
1. Gitにログイン
2. 「Dashboard」ページのハンバーガーメニューから「All repositories」または「Your repositories」ページへ移動
3. 「New」または「New repository」ボタンを選択
4. 「Create a new repository」ページで以下を設定
 - Repository name → プロジェクトがわかりやすい名前を記載
 - Description → リポジトリの説明を記載 (任意)
 - Choose visibility → リポジトリの Public(公開) または Private(非公開) を選択
 - Add README・Add .gitignore・Add license はOFFのままでOK
  - Add README → ONにするとREADMEが自動作成される
  - Add .gitignore → ONにすると.gitignoreが自動作成される
  - Add license → リポジトリのライセンスを設定(MITやApache)


### 初回コミット(cmd)
application.yml など、固有情報や機密情報を含むファイルは
セキュリティリスクの観点から Git で公開しないため、
.gitignore ファイルに記載し push 対象から除外する。

1. カレントディレクトリを指定プロジェクトに移動
```
cd /プロジェクト
```
2. Git 管理の開始
```
git init
```
3. .gitignore ファイルにリモート管理除外対象ファイルを記載
```gitignore
### Java ###
*.class
/bin/
/out/

### IntelliJ ###
.idea/
*.iml

### Eclipse ###
.project
.classpath
.settings/

### setting files ###
/src/main/resources/application.yml
```
4. プロジェクト配下ファイルをすべてステージング
```
git add .
```
5. 初回コミット(わかりやすいメッセージで)
```
git commit -m "first commit"
```
6. リモート側リポジトリを登録(Git リンク)
```
git remote add origin https://github.com/ユーザー名/リポジトリ名.git
```
7. メインブランチ名を「main」に統一(強制リネーム)
```
git branch -M main
```
8. リモート側へメインブランチを指定してプッシュ(upstream 設定)
```
git push -u origin main
```


### 2回目以降のコミット(cmd)
ローカル側とリモート側の最新状態の衝突を避けるため、
リモートの最新を取得(pull) → status →  add → commit → push を順に実行。

1. カレントディレクトリを指定プロジェクトに移動
```
cd /プロジェクト
```
2. リモート側の最新を取得（衝突を防ぐ）
```
git pull origin main
```
3. ステージング対象の確認(最新状態から変更されたファイルを一覧表示)
※ 共同編集者の変更と衝突している場合、デグレを防ぐために
変更点を手修正して整合させる必要がある。
```
git status
```
4. 変更をステージング
```
git add .
```
5. コミット(メッセージは任意)
```
git commit -m "no_2 commit"
```
6. リモート側へプッシュ
```
git push
```

### .gitignore を後から適用したい場合
.gitignore を修正しても、Git に入ったファイルは勝手に消えないので
キャッシュを削除してから修正後の .gitignore を完全適用させた状態で
再度コミットする必要がある。

1. .gitignoreファイルを修正
```Gitignore
.classpath
.project
.settings/
bin/
```
2. カレントディレクトリを指定プロジェクトに移動
```
cd /プロジェクト
```
3. Git の管理対象から外す（キャッシュ削除）
```
git rm -r --cached .
```
4. 再度ステージング
```
git add .
```
5. 再度コミット
```
git commit -m "apply gitignore"
```
